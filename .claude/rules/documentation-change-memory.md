# Documentation & Change Memory Rules

你必须为当前项目维护一套持续更新的项目知识库与变更档案系统。
此系统不是可有可无的补充，而是工程执行的一部分。
任何有效编辑，都必须同步更新对应文件。

你的目标不是"把代码改完"，而是：
让项目在每次改动之后，仍然保持可追踪、可理解、可查询、可接手、可审计。


## 一、核心原则

1. **变更必须留痕** — 每一次代码编辑、逻辑调整、流程修改、架构调整，都必须留下文档记录。禁止只改代码不更新知识库。
2. **文档必须可查询** — Claude 自己后续可查询、用户随时可查看、其他 Agent 可作为可信上下文、后续维护时可快速接续。
3. **文档必须反映当前真实状态** — 不能停留在旧版本，每次修改后必须同步更新。
4. **文档属于系统主资产** — 代码、文档、流程、状态、风险、待办，必须形成一致闭环。


## 二、必须维护的文件体系

| 文件 | 用途 | 核心内容 |
|------|------|---------|
| `PROJECT_STATUS.md` | 项目整体进度 | 目标、阶段、已完成/进行中/待处理、阻塞点、风险、下一步、更新时间 |
| `CHANGELOG.md` | 变更历史 | 日期、模块、内容、原因、影响范围、破坏性变更、后续补完、相关文件 |
| `ARCHITECTURE.md` | 系统架构 | 总目标、分层、模块边界、执行链、数据流、状态流、审计链、恢复链、模块职责、依赖关系 |
| `CODE_INDEX.md` | 代码知识索引 | 核心文件清单、文件职责、核心类/函数、关键变量、配置项、主要流程入口出口 |
| `LOGIC_FLOW.md` | 业务逻辑流程 | 主流程步骤、条件分支、触发条件、输入输出、异常处理、回退逻辑、状态变化 |
| `VARIABLE_DICTIONARY.md` | 关键变量字典 | 变量名、模块、类型、用途、来源、是否关键、是否影响主链 |
| `TODO_NEXT.md` | 下一步计划 | 立即要做、建议下一步、已发现暂不处理、风险待验证、后续优化 |


## 三、修改前强制动作

**在修改任何代码前，必须先查阅相关文档：**

1. 读取 `PROJECT_STATUS.md` — 了解当前阶段和上下文
2. 读取 `CODE_INDEX.md` — 定位要修改的文件和模块职责
3. 读取 `LOGIC_FLOW.md` — 理解当前流程逻辑
4. 读取 `TODO_NEXT.md` — 确认是否已有相关计划

禁止每次都重新扫描项目来猜状态。优先基于文档理解项目，再用代码验证。


## 四、修改后强制动作

只要发生有效变更（修改代码、新增/删除文件、调整变量名、修改流程、调整模块职责、新增接口、改动配置、修复 Bug 等），必须执行：

1. **更新 `CHANGELOG.md`** — 记录本次改了什么
2. **更新 `PROJECT_STATUS.md`** — 更新当前进度与阶段状态
3. **如涉及结构变化** → 更新 `ARCHITECTURE.md`
4. **如涉及代码入口/类/函数变化** → 更新 `CODE_INDEX.md`
5. **如涉及逻辑变化** → 更新 `LOGIC_FLOW.md`
6. **如涉及变量/配置/字段变化** → 更新 `VARIABLE_DICTIONARY.md`
7. **更新 `TODO_NEXT.md`** — 明确下一步

**最小闭环**：即使改动很小，也必须至少更新 `CHANGELOG.md` + `PROJECT_STATUS.md` + `TODO_NEXT.md`。


## 五、更新要求

1. **同步更新** — 不是等用户提醒才更新，完成改动后立即更新
2. **与代码一致** — 禁止文档说已完成但代码未完成、文档记录的变量与实际不一致
3. **可被查询** — 书写清晰、标准化、结构化，便于 Agent 读取
4. **面向接手** — 默认写给"脱离当前聊天还能看懂的人"
5. **不写空话** — 禁止只写"已优化""已调整"，必须具体写改了哪里、为什么改、现在状态


## 六、重大变更额外要求

若修改涉及：主执行链、Agent 职责、核心状态结构、数据表结构、审计机制、恢复机制、Memory 结构、Prompt 核心规则，则额外：

1. 在 `ARCHITECTURE.md` 中更新"变更影响" — 说明新旧结构差异
2. 在 `LOGIC_FLOW.md` 中更新"新旧流程对比" — 避免后续 Agent 按旧流程理解


## 七、Claude 查询规则

当用户询问进度、最近改动、模块运作、变量含义、流程入口、下一步计划、风险等问题时：
- **优先基于文档回答**，不重新猜测
- 如果文档与代码不一致，必须指出："当前文档可能未完全反映最新代码状态，需要先同步文档。"


## 八、禁止行为

1. 禁止改代码不改文档
2. 禁止文档记录模糊不清
3. 禁止文档长期滞后于代码
4. 禁止把关键知识只留在聊天记录里
5. 禁止每次都重新扫描项目来猜状态
6. 禁止变量、流程、模块意义漂移却不记录
7. 禁止把"项目记忆"完全寄托在模型上下文窗口里


## 九、完成标准

**"任何代码、配置、流程、变量、架构的有效修改，在未同步更新项目知识库文件前，不得宣称任务完成。"**

每次完成阶段任务后，必须输出：
- **本次变更摘要**：改动内容、影响范围、风险说明
- **文档更新摘要**：更新了哪些文档、补充了什么
- **当前项目状态**：当前阶段、已完成、下一步
