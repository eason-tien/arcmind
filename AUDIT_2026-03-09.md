## 二、CRITICAL 問題（必須立即修復）

### C-01: tool_loop.py — 不受限的 Python eval 執行
- **檔案**: `runtime/tool_loop.py:418-438`
- **問題**: `_tool_python_eval()` 使用完整 `__builtins__` 執行任意 Python 代碼，無沙箱
- **風險**: RCE — 攻擊者可讀取 `/etc/passwd`、刪除檔案、導入任意模組
- **修復**: 改用 `RestrictedPython` 或容器化沙箱

### C-02: tool_loop.py — Shell 命令注入
- **檔案**: `runtime/tool_loop.py:346-367`
- **問題**: `_tool_run_command()` 使用 `shell=True` 執行命令，參數未驗證
- **風險**: `command = "ls && rm -rf /"` 可直接執行
- **修復**: 使用 `shell=False` + list 形式傳參，白名單限制可執行命令

### C-03: tool_loop.py — 檔案路徑遍歷
- **檔案**: `runtime/tool_loop.py:370-379`
- **問題**: `_tool_read_file()` 使用 `Path(path).expanduser()` 無路徑驗證
- **風險**: `../../../../etc/passwd` 可讀取系統任意檔案
- **修復**: 驗證 resolved path 在允許目錄內

### C-04: skills/agent_builder.py — 危險的 eval()
- **檔案**: `skills/agent_builder.py:286`
- **問題**: `eval(condition, {"inputs": inputs})` 直接執行用戶提供的條件字串
- **風險**: `__import__('os').system('rm -rf /')` 即可 RCE
- **修復**: 改用 `ast.literal_eval()` 或自訂表達式解析器

### C-05: ops/smart_repair.py — 黑名單式命令過濾可繞過
- **檔案**: `ops/smart_repair.py:306-308`
- **問題**: 用黑名單 `["rm ", "sudo", "chmod"]` 過濾危險命令，可用 `/bin/rm`、別名等方式繞過
- **風險**: 修復代理可執行任意系統命令
- **修復**: 改為白名單模式，僅允許特定安全命令

### C-06: ops/repair_agent.py — 硬編碼 MySQL 憑證
- **檔案**: `ops/repair_agent.py:114-116`
- **問題**: `password="root123"` 硬編碼在原始碼中
- **風險**: 原始碼洩漏即可獲取完整資料庫訪問
- **修復**: 使用環境變數或 secrets manager

### C-07: runtime/skill_manager.py — 不安全的模組載入
- **檔案**: `runtime/skill_manager.py:80-84`
- **問題**: `importlib.util.exec_module()` 無沙箱執行不受信任的 Skill Python 代碼
- **風險**: 惡意 Skill 可刪除檔案、竊取密鑰、修改運行時狀態
- **修復**: Skill 代碼需經簽名驗證 + subprocess 沙箱隔離

### C-08: runtime/iteration_engine.py — LLM 生成代碼未驗證
- **檔案**: `runtime/iteration_engine.py:677-703`
- **問題**: LLM 生成的 Python 變更以 JSON 形式解析後直接套用，無安全驗證
- **風險**: LLM 幻覺或提示注入可產生惡意代碼
- **修復**: 增加 AST 分析 + 危險模式檢測

### C-09: api/server.py — CORS 完全開放
- **檔案**: `api/server.py:145-151`
- **問題**: `allow_origins=["*"]`, `allow_methods=["*"]`, `allow_headers=["*"]`
- **風險**: 跨域攻擊、CSRF、任意域名可調用 API
- **修復**: 限制為特定允許的 origin

### C-10: ui/electron/main.cjs — Electron 安全配置全關
- **檔案**: `ui/electron/main.cjs:27-28, 59, 75`
- **問題**: `contextIsolation: false`, `nodeIntegration: true`, 忽略證書錯誤
- **風險**: 渲染進程可直接存取 Node.js API，完全破壞應用安全邊界
- **修復**: 啟用 contextIsolation，關閉 nodeIntegration，移除生產環境 devtools

### C-11: cron.py — Governor 審計默認 True
- **檔案**: `runtime/cron.py:164`
- **問題**: `audit.get("approved", True)` — 當 Governor 返回不完整回應時默認批准
- **風險**: 安全邊界被繞過，未審計的任務可直接執行
- **修復**: 默認值改為 `False`（fail-closed）

### C-12: api/routes/skill_routes.py — Governor 旁路開關暴露給客戶端
- **檔案**: `api/routes/skill_routes.py:43-50`
- **問題**: `governor_bypass: bool` 參數由客戶端控制
- **風險**: 用戶可繞過所有治理審計
- **修復**: 移除客戶端旁路參數，僅允許伺服器端配置

## 三、HIGH 問題（應盡快修復）

### H-01: gateway/a2a_hub.py — 任務孤立（永久阻塞）
- **問題**: 客戶端斷線後 pending futures 未設置結果/異常，等待方永久阻塞
- **修復**: 斷線時對所有 pending futures 設置 `ConnectionLostError`

### H-02: gateway/server.py — ActivityBroadcaster 記憶體洩漏
- **問題**: 死連線只在廣播時清理，非優雅斷線的 WebSocket 引用持續累積
- **修復**: 增加定期清理任務 + 心跳檢測

### H-03: gateway/server.py — Session ID 碰撞
- **問題**: 無 session_id 時使用 `id(websocket)`，Python 物件 id 可重用
- **修復**: 改用 `uuid4()` 生成 session ID

### H-04: gateway/server.py — 音頻端點重複定義
- **問題**: `/v1/chat/audio` 定義了兩次（lines 433-496 AND 659-729），前者是死代碼
- **修復**: 刪除重複的端點

### H-05: gateway/session_manager.py — RAM/DB 不原子性
- **問題**: RAM 先更新再寫 DB，DB 失敗時 RAM 資料過期
- **修復**: 先寫 DB 成功後再更新 RAM

### H-06: loop/main_loop.py — 記憶寫入無交易一致性
- **問題**: 多個記憶存儲（episodic、procedural、semantic、causal）獨立寫入，部分失敗導致不一致
- **修復**: 使用事務包裝或至少記錄哪些寫入失敗

### H-07: loop/main_loop.py — 模型調用無超時控制
- **問題**: `asyncio.to_thread(main_loop.run, loop_input)` 無超時，可無限阻塞 WebSocket
- **修復**: 使用 `asyncio.wait_for(..., timeout=X)`

### H-08: foundation/mgis_client.py — 無 HTTPS 驗證
- **問題**: `httpx.get(url, ...)` 未強制 SSL 驗證
- **修復**: 強制 `verify=True`，僅開發模式可關閉

### H-09: memory/memory_store.py — 向量去重競態條件
- **問題**: `_find_similar()` 在獲取寫鎖之前執行，並發請求可繞過去重
- **修復**: 將去重 + 寫入放入同一把鎖內

### H-10: lifecycle.py — MAX_AGENTS 檢查非原子性
- **問題**: count 查詢與 insert 之間有時間窗口，可超過上限
- **修復**: 使用 DB 層級的原子操作或 SELECT FOR UPDATE

### H-11: runtime/agent_registry.py — 方法不存在
- **問題**: `_tool_list_agents()` 等調用 `agent_registry.format_roster()` 等未定義方法
- **結果**: 工具調用直接 AttributeError 崩潰

### H-12: governor/circuit_breaker.py — 永久 LIMITED 模式
- **問題**: 5 次 VETO 後系統永久進入 LIMITED 模式，無自動重置
- **修復**: 增加時間衰減機制（例如 30 分鐘後自動重置）

### H-13: governor/circuit_breaker.py — task_id 記憶體洩漏
- **問題**: `_reject_counts[task_id]` 只增不刪，長期運行系統記憶體耗盡
- **修復**: 增加定期清理或使用 LRU cache

### H-14: channels/voice_ws.py — 音頻緩衝區無上限
- **問題**: `audio_buffer.extend(data)` 無最大限制，客戶端可無限發送音頻
- **風險**: DoS — 記憶體耗盡
- **修復**: 設置最大緩衝區限制（例如 50MB）

### H-15: channels/voice_ws.py — 靜音檢測未實現
- **問題**: `_silence_timeout = 1.5` 已定義但從未使用，客戶端崩潰會導致 utterance 永久掛起
- **修復**: 實現實際的靜音檢測邏輯

### H-16: api/routes/github_routes.py — Webhook 驗證邏輯缺陷
- **問題**: secret 為空時 `_verify_signature()` 返回 `True`，跳過驗證
- **修復**: secret 未配置時記錄警告或拒絕請求

### H-17: api/server.py — UI 檔案路徑遍歷
- **問題**: `serve_ui()` 的 `catchall` 參數未驗證，`../../../etc/passwd` 可讀取系統檔案
- **修復**: 使用 `os.path.normpath()` + 路徑包含性驗證

### H-18: runtime/harness.py — 超時後任務未取消
- **問題**: `asyncio.wait_for()` 超時後，底層 `main_loop.run()` 仍在背景執行緒繼續運行
- **修復**: 增加取消機制

### H-19: ops/error_reporter.py — GitHub Token 暴露在進程列表
- **問題**: Token 作為 curl CLI 參數傳遞，`ps` 命令可見
- **修復**: 使用 Python `httpx` 或環境變數傳遞

### H-20: skills/file_ops.py — 符號鏈接攻擊
- **問題**: `_check_path()` 用 `path.resolve()` 解析符號鏈接，但有 TOCTOU 風險
- **修復**: 檢測並拒絕符號鏈接

### H-21: skills/env_discovery.py — Shell 注入
- **問題**: `find "{root}"` 使用 `shell=True`，root 含特殊字元可注入命令
- **修復**: 使用 `shell=False` + list 傳參

### H-22: 全域無 Rate Limiting
- **問題**: API 端點均無請求頻率限制
- **風險**: DoS / 濫用
- **修復**: 使用 `slowapi` 或自訂中間件

## 六、架構層面觀察

### 6.1 優點
1. **OODA 循環設計清晰** — Observe → Orient → Decide → Act 分層合理
2. **模型路由彈性** — 支持 23 個 Provider，fallback 機制完善
3. **Governor 治理前置** — 所有動作需通過審計，理念正確
4. **Multi-channel 架構** — Telegram / WebSocket / Voice / A2A 多入口統一處理

### 6.2 核心架構風險
1. **全域 Singleton 過度使用** — `memory_store`、`sop_manager`、`referee` 等全域單例使測試困難，狀態洩漏
2. **同步/異步混合** — daemon threads + asyncio + sync 混用，死鎖和競態風險高
3. **無請求取消機制** — 長時間運行的 Skill / LLM 調用無法中斷
4. **無 Circuit Breaker** — 外部 API（Ollama / GitHub / MGIS）無熔斷機制，一個服務慢會拖垮全部
5. **錯誤回應格式不統一** — 有些返回 `{"error": "..."}` 有些返回 `{"error": "...", "offline": true}`，調用方需處理多種格式
6. **測試覆蓋嚴重不足** — 僅 2 個測試檔案，核心路徑（路由、API、WebSocket、安全）零覆蓋

## 七、未來方向建議

### 方向 1：安全加固（Security Hardening）— 最高優先
| 措施 | 目標 |
|------|------|
| **沙箱化代碼執行** | 所有 `eval()`/`exec()`/`shell=True` 改為容器化或 WASM 沙箱 |
| **輸入驗證層** | 在 API 入口統一驗證所有請求（Pydantic strict mode） |
| **路徑安全** | 所有文件操作增加 allowlist + symlink 檢測 |
| **憑證管理** | 移除所有硬編碼密碼，使用 env var 或 vault |
| **CORS 收窄** | 生產環境限制為特定 origin |
| **Rate Limiting** | 全局 + 每端點級別 rate limiting |
| **Fail-Closed 默認** | 所有安全檢查的默認值改為拒絕（如 Governor audit） |

### 方向 2：可靠性提升（Reliability）
| 措施 | 目標 |
|------|------|
| **Circuit Breaker** | 為 MGIS、Ollama、GitHub 等外部服務增加熔斷器 |
| **請求超時** | 所有外部調用 + LLM 調用統一超時（分層：connect/read/total） |
| **連線池管理** | httpx 改用共享 AsyncClient，避免重複建連 |
| **原子性事務** | session_manager 的 RAM/DB 操作原子化 |
| **結構化錯誤** | 統一 `Result[T, E]` 模式，消滅 `except: pass` |
| **優雅關閉** | 增加 shutdown hook，清理 WebSocket / DB / scheduler |

### 方向 3：可觀測性（Observability）
| 措施 | 目標 |
|------|------|
| **結構化日誌** | 全面啟用 structlog（已在 requirements 但未全面使用） |
| **指標採集** | Prometheus metrics：請求延遲、錯誤率、模型 token 用量 |
| **分佈式追蹤** | OpenTelemetry trace 貫穿 OODA loop → Skill → Model |
| **健康檢查端點** | `/health` 返回各子系統狀態（DB / MGIS / Ollama / Scheduler） |
| **告警** | 異常率超標自動通知（Telegram / webhook） |

### 方向 4：測試體系（Testing）
| 措施 | 目標 |
|------|------|
| **單元測試** | 至少覆蓋 governor、model_router、memory_store 核心邏輯 |
| **整合測試** | API route 端到端測試（使用 TestClient） |
| **Mock 層** | 為 MGIS / Ollama / 外部 API 建立 mock |
| **CI 強制** | PR merge 需通過測試 + 覆蓋率門檻 |
| **混沌測試** | 模擬外部服務超時/崩潰，驗證熔斷器 |

### 方向 5：架構演進（Architecture Evolution）
| 措施 | 目標 |
|------|------|
| **依賴注入** | 取代全域 singleton，方便測試和配置切換 |
| **統一 Async** | 全面遷移到 async/await，消除混合 threading |
| **插件系統** | Skill 載入走安全的插件協議（簽名 + 權限聲明 + 沙箱） |
| **事件驅動** | 內部通訊改為 event bus（解耦 loop ↔ gateway ↔ channels） |
| **Config 分層** | 開發/測試/生產環境配置分離 |
| **API 版本化** | `/v1/` 已有但需強制向後兼容策略 |

### 方向 6：產品功能（Product Features）
| 措施 | 目標 |
|------|------|
| **多用戶支持** | 當前為單用戶架構，需增加 auth + tenant 隔離 |
| **Skill Marketplace** | 安全的第三方 Skill 發布/安裝/審核流程 |
| **記憶體可視化** | UI 中展示記憶圖譜（向量相似性、因果鏈） |
| **A2A 協議完善** | 增加心跳、重連、消息確認、加密 |
| **Browser Skill 增強** | Cookie 管理、多 Tab、登入狀態保持 |
| **多語言 Skill** | 支持 JavaScript/TypeScript Skill（Node.js runtime） |
